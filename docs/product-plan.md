# 我的食谱产品计划书

> 最后更新：2026-07-08
> 当前状态：内测阶段。核心功能（记录、AI 整理、分类、配图、多人协作）已上线；下一步重点是验证图库直连、扩充数据、补齐安全与协作能力，再向微信小程序延伸。

---

## 1. 产品愿景与目标

“我的食谱”不只是一个记录菜谱的小网页，而是想做成一个大家一起维护的智能菜谱库：朋友、家人、同事看到好吃的做法都能顺手补进去；不知道今天吃什么时可以直接从里面挑；看到做菜视频，也能让 AI 帮忙整理成能照着做的菜谱。

一句话目标：

> 复制一个做菜视频链接，AI 帮你整理成菜谱；大家一起补充，慢慢变成我们自己的美食库。

最终它应该是一个越用越有味道、越填越像家的工具：看到好菜随手收进来、不知道吃什么系统帮你挑、买菜前一键生成清单、做饭时手机打开步骤、家人朋友一起维护、AI 帮忙整理和优化。

- 首页世界大类横向滚动展示（Netflix 式），搜索/筛选态自动回落纵向列表

**智能辅助**
- 右下角 AI 悬浮助手，快捷入口（视频链接、字幕文案、食材推荐、优化菜谱、批量整理），结果以草稿卡片展示并一键填入表单

## 4. 当前不足与风险

> 如实列出现阶段的短板，供判断优先级。多数是内测阶段可接受的取舍，但要心里有数。

### 4.1 技术架构层面

- **无自动化测试**：改动靠手动验证。且当前开发机没有浏览器环境，端到端（选图、Canvas 压缩、图库搜索等依赖浏览器的功能）只能靠人工在浏览器里点，存在验证盲区。
- **前端直连外部服务**：AI、图库都是前端直接调。国内网络下 GitHub Pages 访问本就不稳，Pixabay 服务器又在海外，直连稳定性未经验证。

### 4.2 数据安全与协作

- **密钥明文存前端**：AI 的 API Key、Pixabay 的 Key 都存在浏览器 localStorage，任何人按 F12 打开控制台就能看到。内测自用可接受，但不能对外公开推广。
- **数据无并发保护**：所有人读写的是同一个 JSON 文件，全量覆盖式保存。两个人几乎同时保存时，后保存的会盖掉前一个人的改动，且没有提示。
- **无登录、无追责**：全员可编辑可删除，操作日志只记在本地浏览器，换个人/换台设备就看不到，误删或恶意修改无法追溯到具体的人。

### 4.3 功能体验

- **Base64 图片拖慢同步**：本地选图和图库选图都压缩成 Base64 存进共享 JSON。图片一多，数据文件变大，多人同步变慢，导出/备份文件也变大。这是过渡方案的固有代价。
- **图库不保证精确**：Pixabay 是通用图库，搜“红烧肉”给的是“看起来像这类菜”的照片，不一定是那道菜本身。已做成候选图挑选来缓解，但本质局限还在。
- **AI 依赖公开信息**：视频没有公开字幕时，AI 无法凭空“看”画面，只能靠标题、简介或用户补充整理。

### 4.4 合规与依赖

- **配图来源须合规**：已明确不采用网络爬图（版权风险 + 违背项目授权原则）。合规来源限定为：用户自拍/上传、免费授权图库（Pixabay CC0）、AI 文生图、开源库自带授权图。
- **外部依赖不可控**：GitHub Pages、Pixabay 都是外部服务，一旦访问受限或政策变化会直接影响可用性。

## 5. 合理建议（按投入产出比排序）

1. **优先验证 Pixabay 直连**（零成本，决定生死）
   图库配图是这轮的主推功能，但能不能用完全取决于国内能否直连。建议第一件事就是配个免费 Key 在浏览器里实测。通了功能就成立；不通就得加后端代理。不验证，这个功能等于悬空。

2. **数据保存加防覆盖**（改动小，收益高）
   多人协作最现实的风险是“辛苦补的菜被别人覆盖”。加一个简单的版本号/时间戳校验（保存前比对，发现别人已改就提示而不是硬盖），改动不大，能挡住最常见的数据丢失。

3. **对外推广前，把 Key 移到后端**（安全必做项）
   现在密钥明文在前端，仅限信任的朋友内测可以接受。一旦想公开或扩大范围，必须先把 Key 收到后端代理，否则等于把自己的 AI/图库额度公开送人。

4. **数据扩充优先用开源授权库，别碰爬虫**（合规底线）
   继续扩菜谱是好事，但坚持逐个核对许可证。宁可慢一点人工翻译导入，也不要图快去爬有版权的数据。

5. **暂缓 App 化，先跑通小程序**（避免过早投入）
   原生 App 开发和上架成本高。建议先用网站和小程序验证大家是否真的长期用，再决定要不要做 App。路线见 §8。

6. **建立最小验证习惯**（防回归）
   每次改动后至少手动验证：新增、编辑、保存、AI 面板、配图这几个关键动作能否正常，控制台无红色报错。目前没有自动化测试，人工兜底不能省。

## 6. 关键功能方案

### 6.1 图片配图方案

**背景**：菜谱数据存在阿里云 OSS 的一个 JSON 文件里，文字体积小；图片是大文件，不能直接塞进 JSON，否则数据暴涨、拖慢同步。所以“本地上传图片”必须先解决“图片存哪、怎么拿到网址”。

**三种存储方案对比**

| 方案 | 做法 | 优点 | 缺点 | 定位 |
| --- | --- | --- | --- | --- |
| A：Base64 存进数据 ✅已实现 | 前端压缩后编码成字符串存进 image 字段 | 零成本、纯前端、马上能用 | 数据膨胀约 1.3 倍，图多拖慢同步 | 过渡方案，少量小图 |
| B：阿里云 OSS 上传 ⬜待实现 | 后端加上传接口，图片单独存 OSS 返回 URL | 图文分离不拖慢同步，稳定可长期 | 需改后端 + 配权限，有小额费用 | 图片变多后的目标方案 |
| C：第三方免费图床 💡不推荐 | 接免费图床拿网址 | 不动自己 OSS | 不稳定、可能停服、防盗链 | 仅临时演示 |

**建议路线**：现用方案 A（已实现：本地选图前端 Canvas 压缩，最大边 1280px、单图 ≤300KB，转 Base64，带预览/移除/数量软提示）过渡；图片需求变多后升级方案 B，把历史 Base64 图迁移到 OSS。

**配图来源（三种并存）**

- **从本地选图** ✅：用户自己拍/存的图，压缩后存储。
- **从图库找图（Pixabay）** ✅主推：编辑页“🔍 从图库找图”弹出搜索层，用菜名自动搜真实美食照片，候选图网格点选即用，选中图压缩转 Base64。
  - 为什么选 Pixabay：有中文站、API 支持中文搜索（`lang=zh`）、CC0 免费可商用且无需署名、含大量中国美食图。国内商业图库（站酷/图虫/千图等）多为付费或需授权、缺开放免费 API。
  - 为什么不选 Pexels/Unsplash：只支持英文搜索，中文菜名搜不准；Unsplash 免费额度还很低（50 次/时）。
  - 局限：需用户注册免费 Key（存本机）；通用图库不保证精确（已用候选挑选 + 中→英映射兜底）；**国内直连待实测**（见 §3.1）。
- **AI 生成配图** ✅占位/兜底：本地 Canvas 画按世界大类配色的美化图。因文生图慢且按张收费，退为“图库搜不到/无网络时的兜底”，非主力。代码预留 `IMAGE_GEN_ENABLED` 开关 + `generateImageViaApi` 空壳，未来若确需再接真服务。

### 6.2 AI 视频生成菜谱

**目标**：把 AI 从“辅助聊天”升级成“视频菜谱整理助手”。

**理想流程**：点“AI 生成菜谱” → 粘贴 B 站/抖音/小红书/YouTube 链接 → 系统抓标题/作者/简介/字幕 → AI 整理草稿 → 用户确认补充 → 一键保存。

**已完成**：AI 面板四个快捷入口、粘贴视频入口、B 站字幕抓取、抓取失败字幕粘贴区、草稿卡片、一键填表。

**AI 输出统一结构**：

```json
{
  "name": "菜名",
  "source": "来源 / 作者 / UP主",
  "variant": "做法版本",
  "description": "菜品介绍",
  "ingredients": [
    { "name": "食材名", "amount": "用量", "category": "主料 / 辅料 / 调味料", "note": "小白提示" }
  ],
  "cost": "预计成本",
  "occasion": "适合时间",
  "time": "预计耗时",
  "video": "视频链接",
  "steps": ["步骤1", "步骤2"]
}
```

**后续增强方向** 💡：小红书文案、抖音标题文案、YouTube 字幕、上传 .srt、图片识别菜谱、语音输入、自动估算成本耗时。

**须告知用户**：视频无公开字幕时，AI 不能看画面，只能靠标题/简介/字幕/补充信息整理。

### 6.3 多人协作与权限

**当前内测模式**：全员可编辑，不需注册，朋友点开就能补菜谱，收集最快。风险是任何人能删改、暂不追责、误删恢复不便（缓解措施见 §2 协作安全、隐患见 §4.2）。

**正式权限模型**（未来）：

| 成员 | 添加、编辑 |

## 7. 技术架构

**当前架构**（适合国内小范围内测，成本低、上线快）：

```text
GitHub Pages
    ↓
HTML / CSS / JavaScript
    ↓
阿里云函数计算 HTTP API
    ↓
阿里云 OSS JSON 数据文件
    ↓
AI API / Pixabay 图库
```

**中期架构**（需要真正后端时）：

```text
Web / 小程序 / App
    ↓
后端 API
    ↓
数据库
    ↓
AI 服务 / 视频字幕解析 / 图片搜索代理
```

后端可负责：AI/图库 Key 保护、视频字幕抓取、图片搜索代理、登录鉴权、权限管理、数据备份、操作日志、多端统一接口。

## 8. 平台演进路线

不建议现在直接做原生 App。更稳的路线是逐级验证：

```text
Web 高质量版  →  PWA 手机网页版  →  微信小程序  →  React Native / Flutter App  →  iOS / Android 上架
```

**微信小程序（中期重点）**：菜谱非常适合小程序——家人朋友易打开、微信群易分享、做饭时手机看方便、免下载、微信登录适合协作。
- 第一版功能：首页列表、详情、新增、编辑、AI 生成、今天吃什么、购物清单、分享。
- 注意事项：AI Key 必须后端代理（不能放前端）；视频解析也放后端（小程序跨域更严）；上架前需备隐私政策、用户协议、数据删除说明、AI 内容说明。

**App 化（远期）**：等网站和小程序验证成功后再评估用户量与留存，选 Flutter 或 React Native，备应用市场材料，完善账号与隐私合规。可加：离线菜谱、拍照识别食材、语音输入、家庭共享、做饭计时器、每周菜单、营养分析。
