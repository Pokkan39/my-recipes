# 我的食谱产品计划书

## 1. 我们想做什么

“我的食谱”不只是一个记录菜谱的小网页。它最终想做成一个大家一起维护的智能菜谱库：朋友、家人、同事看到好吃的做法，都可以顺手补进去；不知道今天吃什么时，可以直接从里面挑；看到一个做菜视频，也可以让 AI 帮忙整理成真正能照着做的菜谱。

现在它还处在内测阶段，但方向很清楚：先把网页做扎实，再做成手机上更好用的版本，后续再考虑微信小程序、iOS 和 Android。

一句话目标：

> 复制一个做菜视频链接，AI 帮你整理成菜谱；大家一起补充，慢慢变成我们自己的美食库。

## 2. 目前已经有的能力

当前网站已经具备这些基础功能：

- 菜谱新增、编辑、删除
- 记录菜名、来源、做法版本、介绍、食材、步骤、成本、耗时、适合时间、所属世界、添加者、更新时间
- 多人共同编辑，数据同步到阿里云函数计算 + OSS
- 世界大类导航（现实中的饭 / 二次元中的饭 / 黑暗料理 / 存在于幻想中的饭 / 外国菜），可自己增删改
- 场景标签细分（早餐、午餐、晚餐、宵夜、快手菜、健身餐、下饭菜），与大类两层叠加过滤
- 右下角 AI 悬浮助手，四个快捷入口（视频链接、字幕文案、食材推荐、优化菜谱），结果以草稿卡片展示
- AI 辅助整理菜谱并一键填入表单
- B站链接识别和字幕抓取尝试
- “配一桌菜”按荤素主食数量随机凑一桌
- “我有什么食材”按食材反查能做的菜
- “今天吃什么”随机推荐
- 购物清单汇总
- 我的昵称，记录添加/修改人
- JSON 导入和导出备份
- 内置开源菜谱数据（HowToCook 中式家常菜、json-cookbook 外国菜翻译版），来源已标注
- GitHub Pages 在线访问

这些功能已经能支撑朋友之间小范围内测，UI、AI 体验和分类都已明显完善，接下来重点是继续扩充数据、增强协作安全和向小程序延伸。

## 3. 当前最需要解决的问题

### 3.1 稳定性

上线后的每一次修改都必须保证页面能正常交互，不能再出现脚本错误导致按钮失效的情况。

必须坚持：

- 每次修改 JavaScript 后执行语法检查
- 每次部署后检查控制台是否有红色报错
- 每次上线后至少验证新增、编辑、保存、AI 面板这几个关键动作

### 3.2 AI 体验

现在 AI 更像一个聊天框，用户还需要自己理解该怎么问。后续应该把它改成明确的任务入口：

- 粘贴视频链接生成菜谱
- 粘贴字幕或文案生成菜谱
- 根据现有食材推荐菜
- 优化已有菜谱

用户不应该研究 AI 怎么用，而应该点一个按钮就知道下一步该干什么。

### 3.3 美食氛围

现在页面更像一个管理工具，不够像美食网站。后续 UI 应该更有食欲感，更适合分享给别人。

需要突出：

- 菜品图片
- 今日推荐
- 食材标签
- 耗时、成本、适合场景
- 温暖、家常、有烟火气的视觉风格

## 4. A 计划：AI 视频生成菜谱

### 4.1 目标

把 AI 从“辅助聊天”升级成“视频菜谱整理助手”。

理想流程：

1. 用户点击“AI 生成菜谱”
2. 粘贴 B站、抖音、小红书或 YouTube 的视频链接
3. 系统尝试获取标题、作者、简介和字幕
4. AI 根据已有信息整理菜谱草稿
5. 用户确认或补充
6. 一键保存到菜谱库

### 4.2 第一阶段优先支持 B站

第一阶段优先做好 B站，因为目前已经有了基础识别逻辑。

需要完善：

- BV 链接识别
- b23.tv 短链接识别
- 标题、UP主、简介获取
- 字幕列表获取
- 字幕 JSON 解析
- 字幕失败后的人工粘贴引导

需要明确告诉用户：如果视频没有公开字幕，AI 不能凭空看视频画面，只能根据标题、简介、字幕或用户补充的信息整理。

### 4.3 AI 输出结构

AI 最终必须整理出统一格式：

```json
{
  "name": "菜名",
  "source": "来源 / 作者 / UP主",
  "variant": "做法版本",
  "description": "菜品介绍",
  "ingredients": [
    {
      "name": "食材名",
      "amount": "用量",
      "category": "主料 / 辅料 / 调味料",
      "note": "小白提示"
    }
  ],
  "cost": "预计成本",
  "occasion": "适合时间",
  "time": "预计耗时",
  "video": "视频链接",
  "steps": ["步骤1", "步骤2"]
}
```

### 4.4 AI 面板改造

AI 面板打开后不要只显示聊天输入框，而要先给用户几个明确入口：

- 粘贴视频链接生成菜谱
- 粘贴字幕或文案生成菜谱
- 根据食材推荐一道菜
- 帮我优化当前菜谱

生成结果也不要只输出一段 JSON，而要显示成菜谱草稿卡片：

```text
AI 已整理出菜谱：

菜名：蒜香鸡翅
耗时：约30分钟
成本：约25元
主料：鸡翅、蒜
调味料：生抽、蚝油、蜂蜜

[一键填入表单] [继续补充] [重新生成]
```

### 4.5 后续增强方向

后续可以继续支持：

- 小红书文案粘贴
- 抖音视频标题和文案粘贴
- YouTube 字幕
- 上传 .srt 字幕文件
- 图片识别菜谱
- 语音输入菜谱
- 自动生成购物清单
- 自动估算成本和耗时

```text

    ↓

    ↓
```

## 9.5 图片上传方案（规划中，暂未实现）

### 背景与难点

目前菜谱的"配图"只能填图片网址（URL），没有"从本地选图上传"的能力。原因是当前架构里菜谱数据存在阿里云 OSS 的一个 JSON 文件中，文字体积很小；而图片是大文件（一张手机照片常有几 MB），不能塞进这个 JSON，否则数据文件会暴涨、拖慢多人同步。所以要做"本地上传图片"，必须先解决"图片存哪、怎么拿到可访问网址"的问题。

### 三种方案对比

**方案 A：图片转 Base64 存进菜谱数据（零成本，临时用）**
- 做法：前端把图片压缩后编码成 Base64 字符串，直接存进菜谱的 image 字段。
- 优点：不需要任何额外服务和费用，改动只在前端，马上能用。
- 缺点：Base64 会让数据体积膨胀约 1.3 倍，几张图就能让共享 JSON 涨到几 MB，多人同步越来越慢，规模大了会拖垮整个共享库。
- 适用：仅适合偶尔配少量小图，必须加严格大小限制（如单图压缩到 200-500KB 以内）和总量控制。

**方案 B：阿里云 OSS 图片上传（推荐，正规做法）**
- 做法：给阿里云函数增加一个图片上传接口，图片单独存进 OSS（配公共读或签名访问），返回图片 URL 填进菜谱 image 字段。
- 优点：图片与数据分离，不拖慢同步；稳定、可长期使用；是正经的生产做法。
- 缺点：需要改后端函数、配置 OSS 访问权限，且 OSS 存储和流量会产生小额费用（量小时很便宜，但非零）；需要在阿里云控制台配合操作。
- 适用：图片较多、要长期稳定时的目标方案。

**方案 C：第三方免费图床（折中）**
- 做法：接入免费图床服务，上传后拿网址。
- 优点：不动自己的 OSS，可能免费。
- 缺点：免费图床不稳定、可能停服跑路、可能有防盗链，图片随时可能失效；不适合长期。

### 建议路线

先按需选择：偶尔配少量小图用方案 A（零成本、加大小限制）过渡；等图片需求变多、要求稳定时，升级到方案 B（阿里云 OSS 上传），把 Base64 存的历史小图逐步迁移到 OSS。方案 C 只在临时演示时考虑。

### 待实现清单

- 编辑表单增加"本地选图上传"入口（与现有"填图片网址"并存）
- 上传前前端压缩、限制大小和格式
- 方案 B 需：新增 OSS 上传接口、配置访问权限、返回 URL、失败降级提示
- 图片加载失败时的占位处理

## 12. 最终愿景

最终它应该变成一个大家愿意长期使用的智能菜谱工具：

- 看到好菜，随手收进来
- 不知道吃什么，系统帮你挑
- 买菜前，一键生成清单
- 做饭时，手机打开步骤
- 家人朋友一起维护
- AI 帮忙整理和优化

它不是一个冷冰冰的管理系统，而是一个越用越有味道、越填越像家的美食库。
